# План реализации ChatList

Поэтапный порядок работ по спецификации [PROJECT.md](PROJECT.md).

---

## Этап 1. Подготовка проекта

1. Создать структуру модулей:
   - `db.py` — работа с SQLite
   - `models.py` — конфигурация и логика нейросетей
   - `network.py` — HTTP-запросы к API
   - `main.py` — GUI и точка входа
2. Обновить `requirements.txt`: PyQt6, httpx (или requests), python-dotenv.
3. Добавить `.env.example` с шаблоном переменных для API-ключей.
4. Реализовать инициализацию БД при первом запуске (создание таблиц, начальные настройки).

**Результат:** проект запускается, БД создаётся автоматически, зависимости установлены.

---

## Этап 2. Модуль базы данных (`db.py`)

1. Подключение к SQLite (один файл, например `chatlist.db`).
2. CRUD для таблицы `prompts` (создание, чтение, поиск по тексту и тегам).
3. CRUD для таблицы `models` (список, добавление, редактирование, активация/деактивация).
4. CRUD для таблицы `results` (сохранение выбранных строк, чтение истории).
5. CRUD для таблицы `settings` (чтение/запись пар ключ–значение).
6. Метод получения активных моделей: `get_active_models()`.

**Результат:** все операции с БД доступны через единый модуль, без прямого SQL в GUI.

---

## Этап 3. Модуль нейросетей (`models.py`)

1. Dataclass или класс `Model` с полями из БД.
2. Загрузка списка моделей из `db.py`.
3. Получение API-ключа по имени переменной окружения (`api_id_env`) через `python-dotenv`.
4. Фабрика или реестр адаптеров под разные типы API (OpenAI-совместимый формат как базовый).
5. Валидация: модель активна, URL задан, ключ найден в `.env`.

**Результат:** программа знает, к каким моделям и с какими ключами обращаться.

---

## Этап 4. Модуль сети (`network.py`)

1. Асинхронные или синхронные HTTP-запросы к API моделей (httpx).
2. Единый интерфейс: `send_prompt(model, prompt_text) -> str`.
3. Обработка ошибок: таймаут, HTTP-ошибки, неверный ключ — возвращать текст ошибки вместо ответа.
4. Поддержка OpenAI-совместимого API (`/v1/chat/completions`) как первого адаптера.
5. (Опционально) Адаптеры для DeepSeek, Groq и др.

**Результат:** промт отправляется во все активные модели, ответ или ошибка возвращаются вызывающему коду.

---

## Этап 5. Каркас GUI (`main.py`)

1. Главное окно с зонами:
   - ввод/выбор промта;
   - кнопка «Отправить»;
   - таблица временных результатов;
   - кнопка «Сохранить»;
   - (позже) вкладки или диалоги для настроек, моделей, истории.
2. Модель данных в памяти для временной таблицы результатов (список объектов: модель, ответ, selected).
3. Меню или панель: управление моделями, просмотр сохранённых промтов и результатов.

**Результат:** интерфейс отображается, временная таблица живёт только в памяти.

---

## Этап 6. Ввод и выбор промта

1. Поле ввода нового промта (многострочное).
2. Выпадающий список или диалог выбора сохранённого промта из `prompts`.
3. При выборе сохранённого — подстановка текста и тегов в форму.
4. При отправке нового промта — опциональное сохранение в `prompts` (дата, текст, теги).
5. При вводе нового промта — полная очистка временной таблицы результатов.

**Результат:** пользователь вводит или выбирает промт согласно п. 4.1 PROJECT.md.

---

## Этап 7. Отправка запросов и отображение результатов

1. По нажатию «Отправить»:
   - очистить временную таблицу;
   - получить активные модели из `db.py`;
   - для каждой модели вызвать `network.send_prompt()` (желательно в фоновом потоке, чтобы GUI не зависал);
   - заполнить временную таблицу: название модели, текст ответа, чекбокс `selected`.
2. Таблица на экране: колонки «Модель», «Ответ», «Выбрать».
3. Индикатор загрузки на время ожидания ответов.
4. Обработка частичных ошибок (одна модель упала — остальные показываются).

**Результат:** реализованы п. 4.2 и 4.3 PROJECT.md.

---

## Этап 8. Сохранение выбранных результатов

1. Кнопка «Сохранить»: пройти по временной таблице, отфильтровать строки с `selected = True`.
2. Записать каждую выбранную строку в таблицу `results` (связь с промтом и моделью, текст ответа, дата).
3. Очистить временную таблицу после успешного сохранения.
4. Уведомление пользователю об успехе или об отсутствии выбранных строк.

**Результат:** реализованы п. 4.4 и 4.5 PROJECT.md.

---

## Этап 9. Управление моделями и настройками

1. Диалог/вкладка «Модели»: список, добавление, редактирование, переключатель `is_active`.
2. Поля формы: имя, API URL, имя переменной окружения для ключа, тип API.
3. Диалог настроек через таблицу `settings` (путь к БД, таймаут запросов и т.д.).
4. Просмотр истории сохранённых результатов с фильтрацией.

**Результат:** пользователь может настраивать нейросети без правки кода.

---

## Этап 10. Дополнительные функции (опционально)

1. Поиск и сортировка во всех таблицах GUI (промты, модели, результаты).
2. Экспорт выбранных результатов в Markdown и JSON.
3. Расширение адаптеров API (DeepSeek, Groq и др.).
4. Логирование запросов в файл (модель, промт, статус, время).

**Результат:** улучшенный UX и диагностика, согласно п. 6 PROJECT.md.

---

## Этап 11. Сборка и документация

1. Проверить запуск через `python main.py` и через собранный `ChatList.exe` (PyInstaller).
2. Обновить `README.md`: установка, `.env`, первый запуск, добавление моделей.
3. Ручное тестирование полного цикла: промт → отправка → выбор → сохранение → новый промт.

**Результат:** программа готова к использованию, документация актуальна.

## Этап 12. AI-ассистент для улучшения промтов

Реализация п. 7 [PROJECT.md](PROJECT.md): помощник, который улучшает промт через одну выбранную модель OpenRouter.

### 12.1. Настройки и конфигурация

1. Добавить в таблицу `settings` ключи:
   - `assistant_model_id` — ID модели из таблицы `models` для улучшения промтов;
   - `assistant_enabled` — вкл/выкл функции (опционально).
2. В диалоге **Настройки → Параметры** (или отдельной вкладке «Ассистент»):
   - выпадающий список моделей (одна активная модель OpenRouter);
   - подсказка: «Модель для улучшения промтов, не для массовой отправки».
3. Значение по умолчанию: первая активная модель или `GPT-4o Mini`, если есть в БД.

**Результат:** пользователь может указать, какая модель улучшает промты.

---

### 12.2. Модуль логики ассистента

1. Создать модуль `prompt_assistant.py` (или расширить `network.py`):
   - функция `improve_prompt(model: AIModel, text: str, goal: str | None) -> AssistantResult`;
   - dataclass `AssistantResult`: `original`, `improved`, `alternatives: list[str]`, `adaptations: dict[str, str]` (ключи: `code`, `analysis`, `creative`).
2. Сформировать **system prompt** для модели: просить ответ в JSON с полями `improved`, `alternatives` (2–3 строки), `adaptations` (объект с ключами `code`, `analysis`, `creative`).
3. Вызов через существующий `network.send_prompt()` / OpenRouter-клиент к **одной** модели из настроек.
4. Парсинг ответа:
   - попытка `json.loads()` из текста ответа;
   - fallback: если JSON не распознан — показать сырой текст как «улучшенный» вариант, альтернативы пустые.
5. Обработка ошибок: пустой промт, модель не настроена, HTTP/429/404 — понятные сообщения в UI.

**Результат:** из исходного текста получается структурированный результат для отображения.

---

### 12.3. Фоновый запрос (без блокировки GUI)

1. Создать `ImprovePromptWorker(QThread)` по аналогии с `SendPromptsWorker`:
   - вход: текст промта, модель ассистента, опционально цель (`goal`: «код», «анализ», «креатив»);
   - выход: сигнал `finished(AssistantResult)` или `error(str)`.
2. **Не передавать** объект `Database` в worker-поток (только примитивы и `AIModel` с уже загруженным ключом).
3. Индикатор загрузки («Улучшаем промт…») на время запроса, кнопка «Улучшить промт» блокируется.

**Результат:** интерфейс не зависает при обращении к API.

---

### 12.4. UI: кнопка и диалог результатов

1. В главном окне (`main.py`) рядом с полем ввода промта:
   - кнопка **«Улучшить промт»**;
   - (опционально) выпадающий список цели: «Общее», «Код», «Анализ», «Креатив».
2. По нажатию:
   - взять текст из `prompt_edit`;
   - если пуст — предупредить;
   - запустить worker, по завершении открыть `PromptAssistantDialog`.
3. Диалог `PromptAssistantDialog` (в `dialogs.py`):
   - блок **«Исходный промт»** — только чтение;
   - блок **«Улучшенный промт»** — с кнопкой «Подставить»;
   - блок **«Альтернативы»** (2–3) — список или вкладки, у каждого «Подставить»;
   - блок **«Адаптации»** (код / анализ / креатив) — если модель вернула, с кнопками «Подставить»;
   - кнопка **«Закрыть»**.
4. «Подставить» — записать выбранный текст в `prompt_edit` главного окна, закрыть диалог (или оставить открытым — на выбор при реализации).

**Результат:** пользователь видит варианты и одним кликом подставляет нужный в поле ввода.

---

### 12.5. Интеграция с рабочим процессом

1. После подстановки улучшенного промта:
   - сбросить `_current_prompt_id` (это новый текст, не сохранённый промт);
   - очистить временную таблицу результатов (как при ручном изменении промта).
2. Опционально: чекбокс «Сохранить улучшенный промт в БД» при подстановке.
3. Не отправлять автоматически в все модели — только улучшение; отправка по кнопке «Отправить» как сейчас.

**Результат:** ассистент встроен в существующий цикл без ломки п. 4.1–4.5 PROJECT.md.

---

### 12.6. Тестирование и документация

1. Ручные сценарии:
   - короткий промт → улучшенный + альтернативы;
   - длинный промт;
   - модель не настроена / ошибка API;
   - подстановка в поле и последующая «Отправить».
2. Обновить `README.md`: раздел «AI-ассистент», выбор модели в настройках.
3. (Опционально) логирование запросов ассистента через существующий `log_requests`.

**Результат:** функция документирована и проверена.
